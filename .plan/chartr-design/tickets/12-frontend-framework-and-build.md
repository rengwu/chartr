# Frontend framework and build

## Question

xterm.js is an npm dependency, so a build step exists whether or not anyone wanted one, and wayfinder-maps' no-build, vanilla-JS-inlined-by-Go ethos does not survive (ADR 0006). Given that it is already lost, choose the frontend stack deliberately rather than by accident.

- **Framework, or not?** The cockpit is live, stateful and multiplexed — panes, terminals, pushed map state — which is a real step up from a read-only viewer. Svelte is the obvious light choice; plain TypeScript over the build we already need is the obvious austere one. The question is whether this UI's state is complicated enough to buy a framework, or whether the complexity is mostly *terminals*, which xterm.js already owns.
- **The star-map is canvas** and framework-agnostic, and should port largely as-is (ADR 0001). Whatever is chosen must not fight that — and must not tempt anyone into rewriting the renderer as components, which would throw away the one part of the UI that is already designed and proven.
- **How the frontend reaches the Go binary:** embedded in it (`embed.FS`, as wayfinder-maps does) so distribution stays a single file, or served separately? Ticket 13 hangs on the answer.
- **Websocket state sync:** hand-rolled or a library. Terminal streams and map pushes have very different shapes — one is a firehose of bytes, the other is a small derived model — and pretending they are one transport may be a mistake.

## Answer

**Svelte 5 over Vite, as a plain SPA — no SvelteKit.** The honest assessment of the state: the two hard surfaces are already owned — terminals by xterm.js, the star-map by the ported canvas renderer — but the *chrome* around them is a dozen small live views (sidebar rows with liveness badges, session tabs, the needs-you queue, the review brief, the ticket pane, config views), every one reacting to server-pushed state. That is exactly the size at which hand-written DOM mutation becomes the bug farm, and exactly the shape Svelte's runes fit: the pushed model lands in `$state`, views react, and the compiler emits vanilla DOM code with no virtual-DOM layer to fight the imperative islands. Plain TypeScript was the austere contender and loses only on that chrome; React-family loses on runtime weight and ref/effect discipline around islands for no gain here. SvelteKit is refused — no routing, no SSR, nothing for it to do. TypeScript throughout.

**The islands and the seam (ADR 0010).** Terminals and the star-map are **imperative islands**: Svelte owns the DOM *around* them and hands each a bare element; inside, the island's own code runs and Svelte never reaches in. The star-map's seam is narrow — mount, receive the pushed model, emit "ticket selected" — which is what keeps the renderer one coherent imperative program instead of dissolving into components, the failure mode this ticket was written to prevent.

**The star-map is reimplemented cleanly, not line-ported.** [ADR 0001](../../../docs/adr/0001-build-fresh-reusing-the-model-layer.md)'s "copy freely where it elevates the result" is a *permission*, not an obligation — the red line is only: canvas, one imperative renderer, the designed behaviours of `starmap-design.md` (deterministic layout, camera easing, spatial memory). The chartr's renderer diverges too far from wayfinder-maps' for a line-port to be the clean start it sounds like — pushed state instead of polling, [ticket 09](./09-rendering-chartr-states-on-the-star-map.md)'s moons/beacon/ticker, [ticket 08](./08-the-cockpit-layout.md)'s pane-aware camera — so it is written fresh in TypeScript with three references open: the wayfinder-maps renderer (~880 lines of vanilla ES modules), `starmap-design.md`, and this map's prototype assets. The named risk is **feel-drift** — the proven part is partly tuned constants (easing curves, zoom coupling, starfield parallax, dpr handling) that a rewrite silently drops — and the mitigation is to crib those constants and algorithms directly, which is ADR 0001's copying-where-it-elevates applied at the right grain. ADR 0001 stands: the Go model layer is reused as-is, and the renderer's *design* is still lifted; only the lines are new.

**Delivery: `go:embed` the Vite `dist/` — distribution stays one self-contained binary**, which is the premise [ticket 13](./13-shipping-the-chartr-cross-platform.md) hangs on, and the same split wayfinder-maps proves (it embeds source; the chartr embeds build output, the only difference the build step forces). Development is the standard two-process loop: Vite's dev server serves the frontend with HMR and proxies websockets to the running Go backend. No `WAYFINDER_DEV`-style live-dir hatch: Vite already owns that job in dev, and the hackability promise ([ticket 04](./04-the-prompt-library-the-chartr-injects.md)) is about prompts on disk, not rebuilding the UI.

**Transport: two socket kinds, both protocols hand-rolled over a standard Go websocket library** — the ticket's suspicion that one transport is a mistake, confirmed. A **control socket** (one per browser) carries JSON downstream: server-authoritative, pushing the whole derived model as a snapshot on every change — the model is small enough that diff/patch protocols are complexity with no payoff, and reconnect degenerates to "resend snapshot". A **terminal socket** (one per attached terminal) carries raw PTY bytes to xterm.js downstream and keystrokes upstream, with server-side scrollback replayed on attach (ADR 0006's buffering note). No sync library: socket.io-style transports and CRDTs buy reconnection rooms and merge semantics a single-operator local cockpit never needs — the server is the sole authority on state. Operator actions (approve, spawn, abandon) are beneath this map's altitude; the obvious default is plain HTTP request/response, so a failed action surfaces as a response, not a lost message.

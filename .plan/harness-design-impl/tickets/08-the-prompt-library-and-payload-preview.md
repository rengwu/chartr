# The prompt library and payload preview

## Question

The hackable prompt library, live before any PTY exists. Five role prompts plus the common core, vendored from the wayfinder skills and recording the upstream commit, embedded in the binary and materialized to disk as plain markdown; resolution per role walks space committed config → user config → embedded defaults with `replace` (resets base) and `append` (stacks) semantics per layer; a replacement sitting behind the shipped default is surfaced, never auto-merged. Payload composition assembles core + role prompt + context bundle (map body, ticket, blockers' answers, glossary — ADR 0005, fresh every time) into one markdown document; a preview surface shows, for a chosen ticket and role, exactly what a session would be told, with per-part layer provenance. The review payload always carries the ticket's Done-when and the spec, by assembly.

Done when: process-boundary tests cover the resolution matrix (replace/append across all three layers), behind-default surfacing, bundle assembly from fixture tickets, and the review payload provably containing Done-when and spec; the preview renders the composed payload with provenance in the browser; the materialized library is editable on disk and edits show up in the next composition.

## Answer

The prompt library and payload preview land whole, live before any PTY exists. A new `internal/prompt` package owns the library and composition; a read-only preview endpoint and a Svelte modal render exactly what a session would be told. Layout:

- **`internal/prompt/assets/*.md`** — the shipped library, embedded via `go:embed`: a common `core`, five role prompts (`grill`, `prototype`, `research`, `implement`, `review`), and the method `glossary`. Authored adapting the wayfinder method away from Claude-Code-specific conventions; `SourceRepo`/`SourceCommit` record the vendored origin.
- **`internal/prompt/prompt.go`** — materialization and layered resolution. `Materialize` writes the defaults to `<dataDir>/prompts/*.md` (plus a README) on start, never clobbering an operator's edits, so the materialized file *is* the editable base composition reads from. `resolvePart` resolves one part across three layers — shipped built-in ‹ local user ‹ committed workspace — with `replace` (resets the base, discarding the layers below) and `append` (stacks) per layer; the workspace layer resolves last so a committed replacement wins over a local one (the content half of ADR 0009). A `replace` may record `<!-- forked from <hash> -->`; when that hash differs from the shipped default's short content hash, `LibraryWarnings` surfaces it as behind — never auto-merged (story 47) — and the marker line is stripped so meta never reaches the payload.
- **`internal/prompt/compose.go`** — `Compose` assembles the payload: the resolved `core` then role prompt, then a context bundle built fresh (ADR 0005) — glossary, map body, this ticket, and each blocker's answer pulled inline (`AnswerSection`). For the **review** role only it appends two guaranteed parts: the ticket's `Done-when` (extracted, falling back to the whole ticket so the contract is never dropped) and the `spec`. The result carries the parts with per-segment provenance, any warnings, and the single composed markdown document.
- **`internal/server/payload.go` + route in `server.go`** — `GET /api/spaces/{id}/maps/{slug}/tickets/{num}/payload?role=…`: read-only inspection, so a GET; it discovers the map fresh off disk (so an edit shows up with no restart), gathers the blockers from the derived map, and returns the composed `prompt.Payload`. `deriveSpace` folds `prompt.LibraryWarnings` into the space's warnings so a stale fork is visible without opening the preview.
- **`web/src/lib/PayloadPreview.svelte`** (+ `model.ts`, `actions.ts` types/fetch, and the `⧉ payload` trigger threaded `SpacePane → MapCard → DetailPane`) — a modal that, for the selected ticket, previews any role's composed payload: a role picker, behind-default warnings, each part rendered with a provenance chip per segment (built-in / user / workspace / context), and the whole composed document behind an expander.

Against Done-when — every clause is covered by process-boundary tests in `internal/server/payload_test.go` (six tests, all asserting on the HTTP payload and files on disk, none reaching into the package): **(1)** the resolution matrix — `append` stacking across built-in/user/workspace and `replace` resetting the base with workspace winning over user (`TestPromptResolutionMatrix`); **(2)** behind-default surfacing in both the preview and the space snapshot, with the current-hash fork drawing no warning (`TestBehindDefaultSurfaced`); **(3)** bundle assembly from fixture tickets with a blocker's answer inlined and per-part provenance (`TestPayloadComposesWithProvenanceAndBundle`); **(4)** the review payload provably carrying Done-when and spec in its parts and its composed markdown, and *not* for a non-review role (`TestReviewPayloadCarriesDoneWhenAndSpec`); **(5)** the preview rendered with provenance in the browser (`PayloadPreview.svelte`); **(6)** the materialized library editable on disk with the edit composing on the next preview (`TestMaterializedLibraryEditsCompose`). `go vet`, `go test ./...` (server suite green), `svelte-check` (0 errors / 0 warnings), and the Vite build all pass; the whole flow was also exercised against the real binary — review payload parts `core, review, glossary, map, ticket, blocker #01, done-when, spec`, spec resolved from the map's link, and a live edit reflected.

Three decisions for review to weigh:

- **The prompt content is authored, not pulled from a live upstream this session.** The role prompts, core, and glossary are written by adapting the wayfinder method; `SourceCommit` is a placeholder (`vendored-2026-07 (bump on each sync)`) rather than a real `github.com/rengwu/skills` SHA, since that repo was not fetched here. The mechanism the ticket asks for — embed, materialize, record the upstream commit — is in place; the *specific vendored text and its recorded SHA* are the parts a real sync would replace, and the one thing worth an eye.
- **"The spec" is discovered from the map's own link, not a hard-coded path.** `resolveSpec` reads the first `](…spec.md)` link in the map body and resolves it relative to the map directory (an implementation map names its spec in its own body); when there is no link or it does not resolve, the map body stands in as the spec of record so story 53's guarantee still holds. This keeps the harness following wayfinder's conventions rather than wiring a layout, but the fallback-to-map-body path is a judgment call.
- **The preview is offered on any discovered map, classified or not.** It is inspection — no PTY, no commit, no lifecycle — so it is deliberately *not* one of the "session actions" an unclassified map withholds (ADR 0007, story 13). Reading what a session *would* be told before classifying is a feature, not a leak; flagging in case review reads inertness more strictly.

Scope notes: the frontend preview is not under automated test — the one frontend test point is the star-map island seam (spec, Testing Decisions), and the composition it renders is fully covered at the process boundary. The glossary is treated as a single materialized bundle artifact (ADR 0005 lists it as context, not a role prompt), so it is editable but not itself layered with replace/append.

Review payload should carry this Done-when and the spec by assembly (spec, Prompts and payload).

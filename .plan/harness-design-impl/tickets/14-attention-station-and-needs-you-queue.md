# Attention: the action station and the needs-you queue

## Question

Attention at two altitudes, plus the cockpit's keyboard story. The per-map action station: a numbered badge on the map card toggling a drawer of everything actionable on this map — reviews waiting first, then spawnable frontier tickets ranked by how many each unblocks; hovering highlights the star, clicking acts; the badge rides the map's handle when the map is tucked away. Ambient cross-space attention on the sidebar rows: liveness dot, wants-you flag, the gold beacon echo — flagging a row but never re-sorting it. Actionable cross-space attention in the summonable "Needs you" queue: gate-level signals only (a review at the gate, a dead session), ranked reviews-first, one click jumping to that space with its map summoned and the ticket focused; strictly pull, never auto-surfaced. Keyboard-first navigation lands here as a binding constraint: space switching and queue summoning get keys alongside the map's `M`/Esc; and the no-color-only audit covers the sidebar's dots and flags.

Done when: the drawer ranks reviews first then by unblock count against fixtures; the badge follows the map handle when tucked; queue entries are exactly the gate-level signals and jump-to lands focused; the queue never appears without being summoned (asserted); space-switch and queue keys work; every ambient signal carries a non-color channel.

## Proposed Answer

Everything is derived, client-side, from the model tickets 12/13 already push — no backend change. `web/src/lib/attention.ts` is the one new pure module, tested the same way ticket 13's `session.ts` is (`attention.test.ts`, 11 cases, no DOM):

- **`mapActionItems(map)`** — reviews first (`ticket.review` set, oldest ticket number first), then frontier tickets ranked by direct unblock count (`blockedBy` containing the ticket, ties by ticket number), mirroring `internal/server/gate.go`'s `dependents` closure (the post-approve suggestion's own ranking), generalized into a full list. Returns nothing on an unclassified map — there is no spawn affordance to offer there. `mapActionCount`/`spaceActionCount` are the badge counts (the latter summed across every map, for the tucked handle).
- **`needsYouQueue(spaces)`** — reviews across every space, then halted sessions (`terminal.session && !terminal.alive`, the same predicate the sidebar's death-halt buttons already key off), reviews-first. Ambient liveness never promotes into this list.
- **`spaceAttention(space)`** — `'review' | 'halt' | null`, review winning when both hold, for the sidebar row's echo; **`spaceLiveness(space)`** is the separate, weaker ambient signal (working/quiet), independent of attention — a space can be mid-session on one ticket while another ticket's review still waits.

**Wired in:**
- `MapCard.svelte` — a `ListChecks` badge button in the map's own chrome bar opens `ActionStation.svelte` (a Sheet): click a review row to open the hub directly, click a frontier row to spawn straight away (`defaultRole`, lifted out of `DetailPane.svelte` into `model.ts` so both surfaces default identically) and select the new ticket. Hovering a row calls a new `hoverNum` prop threaded to `StarMap.svelte` → a new `StarMap.hover()` seam method → a dashed, dimmer ring in `starmap.ts` (told apart from the solid selection ring by shape, not just weight).
- `SpacePane.svelte` — the existing "Map" toggle carries `spaceActionCount`'s badge whenever the card is tucked away (`!mapShown`).
- `App.svelte` — a `Bell` pip in the sidebar header summons `NeedsYouQueue.svelte` (a Sheet); its rows call `jumpToQueueEntry`, which sets `selectedId` and the `#s&m&t` hash the space's `SpacePane` instance already listens for (ticket 07's deep-link mechanism, reused rather than re-invented) — the linked star seats itself, no new plumbing into the pane's internals. Each space row in the sidebar also carries `spaceAttention`'s flag (`Eye`/primary for a review, `Warning`/destructive for a halt) and `spaceLiveness`'s dot, both additive and never re-sorting the row.
- Keyboard: a shared `isEditingTarget()` guard moved out of `SpacePane.onKey` into `web/src/lib/keys.ts` (both it and the new global handler need the identical "don't steal a keystroke from the PTY/a field/a dialog" check). A new `App.svelte` window listener adds `[`/`]` (cycle spaces in the pushed pinned-then-recency order, never the filtered view) and `q` (toggle the queue) alongside `SpacePane`'s existing `M`/`Esc`; Escape-to-close on both new Sheets comes from the primitive itself, the same way the bindings drawer already works.

**A flagged tension, resolved rather than deviated from silently:** the spec and this ticket both say "the gold beacon echo" for the sidebar, but `docs/design-system.md` is explicit that amber is exempt only *inside* the star-map island and the chrome stays monochrome plus `--destructive`. I read "echo" as the same *meaning*, not the same hue — the wants-you flag is a distinct icon (`Eye` vs `Warning`) on `--primary`/`--destructive`, never gold, which also satisfies the non-color-only Done-when clause more directly than a recolored dot would. Flagging this here rather than quietly amending the design doc or quietly reintroducing amber.

**Verified:** `svelte-check` (0 errors), `vitest run` (56/56, including the 11 new `attention.test.ts` cases), `go vet ./...` and `go test ./...` (unchanged, since there is no backend surface), and `npm run build` (built CSS has no new raw hex — no amber). Beyond the unit tests, I built the real binary against a scratch fixture space (a small map with a resolved ticket, two independent frontier tickets — one blocking two others — and, via the same stub-agent technique `internal/harnesstest` uses, a spawned session left to die) and read the live control-socket snapshot: the server-derived `frontier`/`blockedBy` and `terminal.session`/`alive` fields matched exactly what `mapActionItems` and `needsYouQueue` expect, and after the stub session died the snapshot showed precisely the halted-session shape (`session` present, `alive: false`) the queue and the sidebar flag key off. I could not complete a pixel-level browser walk of the drawer/queue/hover UI itself — the Chrome extension wasn't connected in this session — so the Svelte-side rendering is verified by type-checking, the build, and reading the diff, not by eye. That visual pass is worth doing before this is treated as fully proven in the browser.

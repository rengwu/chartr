# Who commits, and how work gets abandoned

## Question

History is linear and there are no branches (ADR 0003), so every commit lands in the space's one working tree and nothing is quarantined anywhere. Decide precisely who writes which commit, and how work is undone.

Known: an implementing session commits its code plus a `## Proposed Answer`, and the harness promotes that to `## Answer` at approval (ADR 0004). Open:

- Is the promotion **its own commit**, or amended into the session's? One resolved ticket could be one clean commit, or an honest two that show proposed-then-blessed. The second is more truthful; the first is more pleasant to read later.
- Agents commit **on their own initiative**, and each does it differently. Can the harness rely on that, constrain it (message format, granularity, "never push"), or must it take commits over entirely — and can it even do that across agnostic agents?
- **Abandon / discard / re-grill** from the human-review hub: `git revert`, `reset` (the ticket's commit is the tip, since sessions serialise), or something gentler? And what happens to the `## Proposed Answer` — deleted, or kept as the record of a rejected attempt that a later session should not repeat?
- A session that **dirties the tree without committing**, or dies leaving work in progress (ticket 02 decides how that is detected; this decides what is done with the mess).
- Does the harness ever **push**, or is the remote strictly the human's business? "Linear history" is only a promise the harness can keep locally.
- **Promotion racing a live session.** Serialisation is per-space for *sessions* (ADR 0003), but the harness itself writes at approval. Nothing yet stops this: ticket A sits `proposed`, an independently unblocked ticket B has a live session, the human approves A — and the harness now edits A's file and commits into a working tree an agent is mid-flight in. Either approving blocks while any session runs in the space, or the promotion write gets its own concurrency story. Say which.

## Answer

**Commit ownership splits, and the harness only ever appends (ADR 0008).** The harness commits, deterministically in code, exactly the lifecycle writes it owns: the **claim** at spawn, the **promotion** at approval, the **rejection demotion** at abandonment. The implementing agent commits its own work plus its `## Proposed Answer`, shaped by prompt convention — message format, granularity, and *never push* travel in every role prompt. Convention is all it can be: the harness cannot stop an agent committing however it likes, so it verifies after the fact and **surfaces violations in the cockpit rather than enforcing anything** — cockpit, not autopilot. Taking over commits entirely was rejected as a fiction (agents commit anyway, so the takeover path must handle agent commits regardless); handing agents the gate write was rejected because promotion is a must-always-be-true, and those belong in code.

**Promotion is its own commit — the honest two, never an amend.** The session's commit shows what the agent proposed; the promotion commit shows that a human blessed it, and when. The harness never rewrites a SHA: amending is a history rewrite by the one component that promised linearity, and it breaks outright if the human committed or pushed in between. Harness-owned commits carry structured trailers (agent, model, role, verdict) — see the audit-trail decision below.

**Abandonment: the hub marks, the human undoes, the harness offers levers.** The abandon action itself is one mechanical, harness-committed edit: `## Proposed Answer` is **demoted to prose** — a `### Rejected — <date>` subsection under the Question, with the human's rejection reason appended — so the ticket derives open again and returns to the frontier, while the record of what was tried and why it failed stays in the one file the next session's context bundle is guaranteed to carry. No second status heading, so ADR 0004's narrow extension of the markdown adapter holds. Undoing the rejected *commits* is the operator's business: the hub offers **optional one-click revert** (and reset, where the session's commits are verifiably still the tip) as levers the human pulls, but the harness never fires them on its own. A rejected attempt the human leaves in history is a truthful history.

**A dirty tree is a badge, not a gate.** When a session dies or is abandoned with uncommitted work, the harness surfaces *tree dirty* alongside the halt and touches nothing. A new session **may spawn into a dirty tree** — the human is trusted to clean up first or to decide the mess is harmless. Accepted openly, against the grain of the deciding rule: the failure this invites is a next agent sweeping the previous ticket's debris into its own commit, attributing A's litter to B. That contamination is the operator's to prevent, and the halt-with-badge is where they are told. **Revisit trigger:** if cross-ticket contamination shows up in real driving — or the moment autopilot exists, where no human stands between the badge and the next spawn — clean-tree-at-spawn becomes the deterministic rule it wanted to be.

**The harness never pushes.** The remote is strictly the human's business, in their own terminal; *never push* is a standing prompt constraint for agents, and an agent that pushes anyway is a surfaced violation, not a preventable one. The harness's linearity guarantee ends at the local repository, and the docs say so.

**The gate write proceeds during a live session, as a narrow write.** Approval never waits: promotion is a **pathspec-limited commit** (`git commit -- <ticket file>`) touching exactly the one file no live session owns — B never writes A's ticket, and B's context bundle was assembled at spawn — and immune to whatever the agent has staged, which kills the real foot-gun (a naive `git commit` sweeping the agent's half-staged work into the gate commit). Git's index/HEAD locking makes the interleaving safe; the residual is a millisecond window where an agent's `git commit -a` sweeps A's promotion edit into its own commit — content stays correct, attribution smears, and the harness notices its own commit came up empty and reports it. Blocking approval on a quiet space was rejected (stalls the hub for the length of a session; independently-unblocked tickets are supposed to be independent), as was deferring the commit (an approved-but-still-proposed map on disk is the dual state ADR 0004 exists to prevent).

**Git is the audit trail; there is nothing else to build.** With claim, work, and verdict each a commit, linear history plus the map already answer *who ran what, when, on which model, and how it ended* — the claim commit opens every session's record, the promotion or demotion commit closes it, and trailers make it greppable. No event store, no harness-owned log: a second history can drift from the first, is invisible to vanilla tools, and starts down the iudex event-sourcing road this map declined at charting. The accepted gap: what never touches git — token spend, ad-hoc shells — is unrecorded, which squares with cost visibility already being best-effort and after-the-fact (ticket 02, ticket 14). *This clears the durable-audit-trail fog patch.*

# Rendering harness states on the star-map

## Question

wayfinder-maps' star-map renders six states — resolved, frontier, claimed, blocked, out-of-scope, undermined — through colour, size, glow and pulse, on the principle that *status is the whole star while type rides in the label*. The harness introduces states that viewer never had: **implementing**, **proposed**, **agent review**, **human review**, plus whatever liveness vocabulary ticket 02 settles (working, stuck, dead).

Decide how they read, and prove it by drawing it. The constraints worth respecting rather than discovering:

- **Deterministic layout must survive.** Positions seed from the ticket data so a ticket stays where you learned it, and the design record is emphatic that this spatial memory is the point. Live state must never move a star.
- The palette is **already spent** on six states. Adding five more invites a christmas tree. Is the new axis colour at all — or something orthogonal, like motion, an orbiting mark, or a ring?
- **Human review is a call to action**, not a status. It probably should not read like the states the harness is merely reporting; it is the one star that wants you.
- The star-map becomes **live** for the first time (`fsnotify` → re-derive → push). What does a ticket *changing state under you* look like, given that resolving already perturbs its neighbourhood — and does a live map break the calm the design record was chasing?

Prototype against a real fixture rather than invented data — `../wayfinder/.plan/viewer-hardening` for a planning map, `../expensif/.plan/export-csv-impl` for an implementation map — and link it as an asset.

Prototype: [09-starmap-harness-states.html](../assets/09-starmap-harness-states.html) — open in a browser; three overlay variants on one ported star-map, cycled with ←/→ or `?variant=`; `?` explains how each answers the ticket. **A "Moons" is canonical after review**; B and C are round one, kept for the record. The base six states keep the shipped viewer's exact palette and motion; each variant proposes a different *orthogonal axis* for the harness states: **A Moons** (the session is a body — an amber mote orbits the star; liveness is its motion: working orbits, quiet crawls, dead freezes grey on a greyed orbit; proposed docks it at the rim; agent review adds a violet counter-orbiter; human review breaks the orbital language with a gold beacon), **B Rings** (the pipeline stage is geometry, extending claimed's ring language — rotating dashed ring implementing, sealed solid proposed, violet counter-ring agent review, gold reticle human review; nothing orbits, the calmest), **C Signals** (nothing added — state rides on the star's own light: breathing, glint, amber↔violet oscillation; human review pings *and dims the rest of the map*). Shared: layout computed once and never again (▸ plays the whole lifecycle — implementing → quiet → proposed → agent review → human review → approval igniting the dependent → a death — and no star ever moves), one flare + one fading ticker line per live change, an edge-of-screen gold chevron when the human-review star is offscreen, and a live legend of all twelve states. Fixture `1` is expensif/export-csv-impl (implementation), `2` wayfinder/viewer-hardening (planning) — real graphs, scripted session overlay; the planning fixture's idle HITL session deliberately shows no quiet badge (ticket 02).

## Answer

**The new axis is not colour — it is company: the session renders as a body orbiting the star, leaving the six-state palette untouched.** Prototyped three ways ([the asset](../assets/09-starmap-harness-states.html); **A "Moons" is canonical**, B "Rings" and C "Signals" kept as round one's record). The insight the moon carries: the harness's new states are facts about the *session around* a ticket, not about the ticket's own standing — the base status already says everything the map format knows (the star stays claimed-amber through the whole pipeline, unresolved by ADR 0004) — so session state belongs on a separate visual layer, an object near the star rather than a repaint of it. A body circling is also what claimed's orbital rings were already gesturing at; the moon makes the session a thing you can literally point at.

**Liveness is the moon's motion, so working/quiet/dead cost no palette at all.** Working orbits with a trail; quiet crawls, dimmed and blinking — visibly a hint, not an alarm; dead freezes grey mid-orbit and greys its whole orbital apparatus. The map only renders what the harness surfaces and acts on nothing (ticket 02), and an idle HITL grilling session shows *no* quiet badge — idleness means opposite things by ticket type, and the non-signal is deliberate. The pipeline reads as the moon's story: **proposed** docks it at the star's rim (the work has landed; nobody is circling); **agent review** adds a smaller violet moon counter-orbiting the docked proposal — violet is the one new hue this design spends, on the adversary, safely distant from the six spent colours.

**Human review deliberately breaks the orbital grammar — it is the only state that is a call to action, so it must not read like one more orbital fact.** The docked moon turns white, the star warms toward gold-white, and ping rings emanate on a beat; when the star is offscreen, a gold chevron at the screen edge keeps calling ("#03 wants you") — attention must not depend on where the camera happens to be. This is the star-map's end of ticket 10's hub: the beacon is the summons, the hub is where it lands.

**Live state never moves a star, and a live map stays calm.** Layout is computed once per map from the ticket data (the shipped viewer's deterministic seed, ported unchanged); a push (`fsnotify` → re-derive) changes only colours and overlays — the prototype plays the entire lifecycle, implementing through approval igniting the dependent through a death, with verified zero star movement. A change under you is one flare (the viewer's existing status-change pulse) plus one fading ticker line, then calm; approval's neighbourhood perturbation is the existing ignition language (edge flow starting, frontier pulse) unchanged. B was passed over because rotation-as-liveness goes illegible at distance and the sealed/dashed ring grammar needs a legend to read; C because states riding on the star's own light collide with the brightness vocabulary the base six already own, and its world-dimming call to action punishes the whole map to promote one star. The prototype is throwaway; the decisions above are the deliverable.

# Shipping the harness cross-platform

## Question

The harness targets macOS, Linux and Windows and is meant to be distributed, not merely run on the author's machine. That was decided (ADR 0006) partly *because* it rules out per-platform native work — so this ticket makes sure the bill was read correctly.

- **What is the artifact?** One Go binary with the frontend embedded, serving a browser? That plus a native webview shell — wayfinder-maps builds both, and its webview path needs cgo? Both, as separate downloads, with different support promises?
- **cgo and the webview are the sharp edge.** A pure-Go, browser-only build cross-compiles trivially. The native shell does not, and that asymmetry should drive the answer rather than be discovered during a release.
- **How do people get it** — `go install`, released binaries, Homebrew, a Claude Code plugin marketplace entry (wayfinder-maps ships one, and there is an irony worth noting in distributing an agent-agnostic tool through one agent's marketplace)?
- **What must already be present.** The agent CLIs are not ours to ship, and a committed workspace config can name one the operator lacks (ticket 05). Is there a doctor command, and what does a cold start look like for someone who has none of them?
- **Windows deserves an explicit answer, not an assumption.** PTYs mean ConPTY; path handling differs; and whether the agent CLIs we target even run there is a question that could quietly demote Windows to unsupported. Better to decide that than to imply it.

## Answer

**The bill was read, and the reading held — after the language itself was put on trial.** The grill opened wider than the ticket asked: with native feel as a stated want, Go's own GUI ecosystem was surveyed (own-drawn toolkits that would forfeit [ticket 12](./12-frontend-framework-and-build.md)'s settled frontend; webview_go bare; Wails v3 still alpha) and full replacements were compared — Electron the most proven home for exactly this app shape, **Rust + Tauri the serious contender**. What decided it for Go, with the costs stated rather than assumed: the "model layer reuse" of ADR 0001/0006 was *sized* at ~800 lines plus its test suite — days to port, cheap to walk away from, ADR 0001's "copy freely" being permission and not a moat (ticket 12's renderer precedent) — so the deciding weight fell elsewhere: **Tauri inverts browser-first** (a local window, where today's binary serves any browser), and it puts **the gnarliest part of this codebase — PTY fan-out, scrollback buffers, websocket broadcast — in async Rust instead of goroutines**, the highest-friction language applied at exactly the wrong spot, while making WebKitGTK/WebView2 rendering variance the harness's own bug surface. Remote driving was explicitly *not* the operator's need; Go won anyway on the core's ergonomics. ADR 0006 stands, re-affirmed with its reasoning now honest.

**The artifact is tiered: one supported thing, everything else best-effort ([ADR 0011](../../../docs/adr/0011-one-supported-artifact-tiered-extras.md)).** *Supported*: the pure-Go browser-serving binary with the Vite `dist/` embedded (ticket 12's premise) — one file per platform, all three OSes cross-compiled from a single CI job, because nothing in it needs cgo. *Best-effort*: the native webview shell, per platform, where the toolchain allows — the wayfinder-maps split re-used down to its liftable build scripts. The cgo asymmetry the ticket warned about becomes the tier boundary instead of a release-day discovery: macOS needs its toolchain, Linux needs WebKitGTK, and Windows's shell is — the one pleasant surprise — cgo-free via `go-webview2`. A shell that fails to build never blocks a release; the supported binary is what "the harness" means.

**Distribution: GitHub releases, and nothing else.** goreleaser builds and checksums the supported binaries; the shells attach as extra assets where they built. Explicitly declined: `go install` (awkward against an embedded `dist/`, devs-only anyway), Homebrew (cheap to add later once releases exist — later can decide), and the Claude Code plugin marketplace — the ticket's noted irony of distributing an agent-agnostic tool through one agent's storefront is resolved by abstention.

**There is no doctor.** The up-front check the ticket asked about already exists as [ticket 05](./05-mapping-roles-to-agents.md)'s two surfaces — the registry badge before spawn, the specific hard-block message at spawn ("review is bound to codex … install it, or set a local override") — and a separate command would be the same facts with more ceremony, surfaced away from the moment of need. Ticket 05's "doctor check" promise resolves to the badges themselves. Cold start with **zero** agent CLIs installed is honest and un-dramatic: registry, maps, star-map and ad-hoc shells all work — the agents are not ours to ship, and only spawn blocks, with the message doubling as the installer's to-do list.

**Windows: best-effort native, decided rather than implied.** The facts that make it feasible: claude runs natively; codex has a first-party native path (labelled experimental); opencode runs; pi unverified — the CLIs do not force a demotion. The one real gap is ADR 0006's own `creack/pty`, which has no Windows support and no merged ConPTY path — so the PTY layer is built on a **cross-platform, ConPTY-capable library (e.g. `aymanbagabas/go-pty`) from day one** (ADR 0006 amended), the cheap-now-expensive-later choice that keeps the session core from ossifying unix-only. The tier is stated in the README: builds cross-compiled and smoke-tested in CI, agents run, nobody drives it daily — and **WSL2 documented as the sure path** for anyone who needs certainty over nativeness. Full parity would be a promise the project can't keep; silence would be the one wrong answer the ticket named.

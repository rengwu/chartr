# Telling a planning map from an implementation map

## Question

The two map kinds have different lifecycles. A planning map's ticket is grilled live with a human and resolves directly. An implementation map's ticket runs `implementing → proposed → [agent review] → [human review] → resolved`. So chartr must know which kind it is looking at **before it offers any action** — and getting it wrong means either gating a conversation that needed no gate, or letting code resolve unreviewed.

How does it tell? The candidate signals are inferential: `to-tickets` writes to `.plan/<slug>-impl/` and marks every ticket `type: task`, and the wayfinder skill's other types (`grilling`, `prototype`, `research`) do not appear in an implementation map. Are those signals load-bearing enough to infer from, or should the kind be **explicit** — declared in the map body, or in the committed workspace config?

Push on the awkward cases and say what happens in each:

- A hand-written implementation map that never went through `to-tickets` and follows none of its conventions.
- A planning map containing a `task` ticket — the wayfinder skill explicitly allows one, as the type that *does* rather than decides.
- A map whose Notes override "plan, don't do" and carry execution into the map itself, which the skill also permits.

Then decide the sharper question underneath: is **kind** a property of the *map*, or is it derived **per ticket**? If per ticket, can one map hold both lifecycles at once — and if it can, is that a feature or a thing to forbid?

## Answer

**Kind is declared, never inferred — an explicit property of the *map*, recorded in committed chartr config, and the signals are demoted to a one-time guess.** The candidate signals (`.plan/<slug>-impl/` suffix, every ticket `type: task`, the Notes prose) are each individually breakable and the ticket's own awkward cases prove it: a hand-written implementation map follows none of them, and wayfinder explicitly permits a `task` ticket on a planning map and a Notes override that carries execution. Inferring kind from those signals at read time means a hand-written implementation map reads as planning and its **code resolves unreviewed** — silently, which is the worst way. So kind is an explicit declaration that deterministic code trusts; the heuristics survive only as a *proposal* offered once, when chartr first meets a map, for a human to confirm. This is the "cockpit, not autopilot" standing preference applied exactly: human judgment once, deterministic code forever after, and never a lifecycle running on a guess (the same "surface, never act on a heuristic" line ticket 02 drew).

**The declaration lives in committed chartr config, keyed by map slug — not in the map body.** The map body would be self-contained and visible in one file, but it is a *second* extension of the wayfinder markdown adapter beyond the single non-resolving heading ADR 0004 grants (`## Proposed Answer`), and `TRACKER-MARKDOWN.md` defines no map-level frontmatter for it to sit in. A vanilla wayfinder tool would then meet a `kind:` the tracker spec never defined. Keeping the declaration in a chartr-owned file that is *committed to the space's repo* leaves wayfinder's format untouched — the same map reads unchanged in a vanilla tool — while the declaration stays shared, versioned, and portable across machines and operators (which a chartr-local, uncommitted registry would not: every fresh clone or teammate would re-classify, and two operators could disagree in silence). The accepted cost: the entry sits apart from the map, so renaming a map's directory dangles its config entry — which resolves into inert-until-classified below, not an error.

**Kind is per *map*, and every ticket on it inherits that one lifecycle — mixed lifecycles on one map are unrepresentable.** This is what `CONTEXT.md` already asserts ("a map is always either a planning map or an implementation map") and it dissolves the three awkward cases rather than special-casing them. A `task` ticket on a *planning* map simply resolves directly, as wayfinder intends — the human is already live in the loop, so there is no gate to miss. A Notes-override planning map is whatever kind the human declared at confirmation; the Notes carrying execution is not a second signal that flips its lifecycle. Deriving kind *per ticket from `type:`* was rejected: it would gate a planning map's lone `task` ticket against a spec that does not exist, and make the stricter frontier rule (a blocker must be *approved*, not merely answered — ADR 0004) apply to some edges and not others on the same map. The review apparatus itself is map-shaped: agent review critiques a proposed ticket "against its Done-when and its spec," and only an implementation map has a spec to critique against. A per-map default with per-ticket overrides was rejected for the same reason — it reintroduces the mixed-lifecycle map as a feature and adds a third adapter extension.

**A map whose kind is not yet declared is inert until a human classifies it.** A freshly discovered map — or one whose config entry dangled after a rename — renders on the star-map and is fully readable, but chartr offers **no session actions** on it until its kind is confirmed. Classification is the one action offered, with the inferred guess (`-impl` suffix ⇒ implementation, all-`task` tickets ⇒ implementation) pre-selected for one keystroke. No lifecycle ever runs on a heuristic; a provisional lifecycle that later flips is exactly the ambiguity the explicit declaration was bought to kill. This needs chartr to *discover* maps under a space's `.plan/` and hold a per-map "unclassified" state — machinery ticket 11 (the space registry) will own, and the first chartr-owned per-map state beyond a space's committed config.

**Graduating a finished planning map into an implementation map is something chartr *notices*, not an action it offers.** You run `to-tickets` yourself — in any session, cockpit-spawned or not — and chartr detects the new `.plan/<slug>-impl/` directory and surfaces it for classification, pre-guessing implementation from the suffix. This needs no machinery beyond the discover-and-classify flow the previous decision already requires (a new map directory must surface for classification *regardless of who created it*), and it keeps chartr a cockpit *over* wayfinder rather than a wrapper that wires a specific wayfinder skill into itself — the boundary the map's Out-of-scope warns against. An explicit "graduate" affordance can be added later as pure sugar: it is just a role that spawns a session with the `to-tickets` prompt, and it would still have to handle maps created outside the cockpit anyway. This resolves the "`to-tickets` from inside the cockpit" fog patch: the answer is *no* — chartr observes, it does not orchestrate the method's skills.

Recorded as [ADR 0007](../../../docs/adr/0007-map-kind-declared-not-inferred.md), because declaring-not-deriving cuts against ADR 0004's everything-is-derived grain and a future reader will need the why.

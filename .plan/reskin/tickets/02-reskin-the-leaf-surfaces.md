# Reskin the leaf surfaces — dialogs, forms, detail, payload

## Question

Rebuild the self-contained surfaces — the ones that don't own the shell layout — onto shadcn-svelte primitives, tokens, IBM Plex, and Phosphor icons, deleting their bespoke `app.css` rules as each lands. In scope: `Modal.svelte` → shadcn **Dialog** (the `RegisterForm` "Add a space" dialog rides on it); `RegisterForm.svelte` → **Input** + **Button** + inline validation (the `.register-*` block); `DetailPane.svelte` → **Card**/**ScrollArea** with **Badge** status pills (`.dp-*`, the rendered-markdown `.dp-md` styles rehomed under a `prose`-like token scale); `PayloadPreview.svelte` → the wide Dialog + **ScrollArea** with per-segment layer provenance as **Badge**s (`.modal-card.wide`, `.payload-*`).

Apply distill on the way through: the hand-rolled `.btn` / `.icon-btn` / `.kind-btn` / `.field-src` variants collapse into the shared Button/Badge primitives with a small, named set of variants — not a bespoke class per site. Emoji/unicode glyphs (× close, ▲ warn, ⚠, the register states) become Phosphor icons. Status colour keeps its meaning: `resolved` → a success-tinted badge, `proposed`/`claimed` → neutral/`--primary` emphasis, `out_of_scope` → muted, problems → `--destructive` — no amber. Every surface reads one primary action.

The behaviour is fixed (map "Out of scope"): the dialog open/close contract, the detail pane's inline-blocker assembly and deep-link selection, and the payload's layer provenance all work exactly as before — only the skin changes. Keep the `Modal`/`Dialog` API compatible with its callers or update them in the same ticket.

Done when: each listed surface renders on shadcn-svelte primitives with tokens, IBM Plex, and Phosphor icons, and its old `app.css` block is deleted; the register/add-space, detail-pane, and payload-preview flows behave identically (open, validate, select, scroll, dismiss); no amber remains on these surfaces; `svelte-check` is clean and the Vite build, `vitest`, `go vet`, and `go test` pass.

## Proposed Answer

All four surfaces are rebuilt on shadcn-svelte primitives, vendored via the CLI (`npx shadcn-svelte add dialog button input card badge scroll-area label`, same manual-vendor path as ticket 01 — the CLI writes files but is not run as a live codegen step against a Kit app) into `web/src/lib/components/ui/`. The CLI's default `dialog-content.svelte` imported `@lucide/svelte`'s `X`; swapped for `phosphor-svelte`'s `X` and the `@lucide/svelte` dependency removed, keeping the map's "icons are Phosphor" decision intact everywhere, including inside vendored primitives. `bits-ui` (the Bits UI headless primitives shadcn-svelte's Svelte port sits on) is a new runtime dependency, filed under `dependencies` alongside the other UI libraries; the CLI's `@internationalized/date` (a Bits UI calendar/date-picker peer we never touch — no date components were added) was pruned as unused. `web/src/lib/utils.ts` gained the three prop-shape helpers (`WithElementRef`, `WithoutChild(ren)`, `WithoutChildrenOrChild`) the vendored primitives are typed against.

**Modal.svelte** — same external contract (`open`, `title`, `onClose`, `wide`, `children`), so `App.svelte`'s "Add a space" call site (ticket 03's territory) is untouched. Internally it wraps `Dialog.Root`/`Dialog.Content`/`Dialog.Header`/`Dialog.Title`; `onOpenChange` reports every dismissal (Esc, backdrop, the primitive's own Phosphor-`X` close button) back through `onClose`, mirroring the old native-`<dialog>` contract exactly. `wide` swaps `sm:max-w-sm` for `sm:max-w-2xl`; the content is capped at `max-h-[85vh]` with the actual scrolling left to whichever child needs it (PayloadPreview's own ScrollArea below), so there's one scroll owner, not two nested ones.

**RegisterForm.svelte** — same layout (headline+copy for `first-run`, a bare input+button row for `inline`) and the same prop API, now on `Input` + `Button`; no restructuring.

**DetailPane.svelte** — the aside keeps its `aria-label` and prop contract for `MapCard.svelte` (ticket 03's shell); internally it's a `bg-card`/`ring-1` surface with a `ScrollArea.Root` body. Status reads as a `Badge`: `resolved` → `default` (solid — the strongest neutral fill reads as "done"), `proposed`/`claimed` → `secondary`, `out_of_scope` → `outline` + muted text, and a missing/unresolved blocker reference → `destructive`. **Flagging per the ticket's own instruction:** "resolved → a success-tinted badge" can't be literally true under the map's settled palette — hue ~107 everywhere plus `--destructive` red is the *only* chromatic token, so there is no green to key a success tint off. I read "success-tinted" as "reads positively," which the solid-primary fill for `resolved` achieves without adding a hue outside the settled palette; flagging in case the operator meant something else. The rendered-markdown blocks (ticket body, blocker answers, map material) now share one `.prose-sm` scale added to `app.css` — a small, token-driven stand-in for a typography plugin, used by both DetailPane and PayloadPreview rather than duplicated per surface (distill). The `⧉`/`×` glyphs are Phosphor `Eye`/`X`.

**PayloadPreview.svelte** — the wide `Modal` body is now `ScrollArea.Root`-scrolled; role selection is a `Button` row (`default` when active, `outline` otherwise) rather than hand-rolled pills; per-segment layer provenance is a `Badge`. Same palette constraint as above: the four layers (`built-in`/`user`/`workspace`/`context`) used to read apart by hue (grey/blue/green/purple), which the monochrome-plus-destructive palette no longer has room for. They're told apart by the four remaining non-destructive `Badge` variants instead — `built-in` → `outline` (lightest touch, the shipped baseline), `workspace` → `secondary`, `user` → `default` (most emphasis — the operator's own override), `context` → `ghost` (assembled fresh, set apart as the odd one out). The `⚠` warnings block drops the old amber tint (no amber anywhere per the map) for a neutral bordered box with a Phosphor `Warning` icon — these are heads-up notes, not the "problem" `--destructive` is reserved for. Flagging both hue-to-weight remappings for the operator to confirm they read as intended; nothing here reopens a settled decision, just fills gaps the four-hue/amber-tint choices left unaddressed once the palette went monochrome.

**Behaviour** (map "Out of scope" / this ticket's own clause): dialog open/close/backdrop/Esc, the detail pane's blocker-answer inline assembly and deep-link ticket selection (`#s=…&m=…&t=…`), and the payload preview's role-switch/re-fetch/raw-document toggle all run through unchanged logic — only the templates and styling moved. Verified live: `go run ./cmd/harness` + `npm run dev`, driving this repo's own `wayfinder-harness — cockpit reskin` map in the cockpit (dogfooding) — selected ticket #02 itself, its `resolved` blocker #01 badge, the blockers/prose rendering, opened the payload preview (role switch, per-segment badges, warnings path present in the code though this ticket produced none to show, raw-document disclosure), and the "Add a space" modal — all via screenshots, no console errors.

**Against Done-when.** Every surface is on primitives + tokens + IBM Plex + Phosphor; there was no legacy `app.css` block left to delete on any of the four (ticket 01's operator call already stripped it — confirmed by re-reading `git show` of the pre-ticket state, only the two components' own scoped `<style>` blocks remained, both now removed). Flows behave identically per the walkthrough above. `grep -i amber` and `grep -i d9a441` against the built CSS return nothing. `npm run check` — 0 errors/0 warnings; `npm run build` — clean (CSS 37.4 kB, no errors); `npm test` — 33/33 passing (unchanged suite — no new unit tests were needed, nothing here has logic beyond markup/props); `go vet ./...` and `go test ./...` — clean.

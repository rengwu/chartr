# Re-theme the islands — xterm and the star-map from the shared tokens

## Question

Point the two imperative islands at the design-system tokens through the seam, so the whole app reads as one system — **without** reaching inside either renderer (ADR 0010). Use the `tokens.ts` bridge from ticket 01 at the mount/model seam only.

**xterm (`Terminal.svelte`):** build the xterm `ITheme` (background, foreground, cursor, selection, and the 16 ANSI slots) from the theme tokens — `--card`/`--background` for the surface, `--foreground` for text, `--muted-foreground` for dim, `--destructive` for red — so the terminal surface matches the reskinned tab strip instead of xterm's defaults. The mapping is computed in the wrapper and handed to the island; the island's internals don't change.

**star-map (`starmap/theme.ts`):** the six status hues are **kept** as exempt categorical data-viz colour (map decision), but re-tuned so they sit legibly on the theme's warm near-black `--card`, and the card's own background (currently the hard-coded `#05070d`) is derived from the token surface rather than a magic constant. Reconcile against ADR 0010's feel-drift risk: this is a palette re-tune at the seam, not a renderer change — the star sizes/glows/pulse and the tuned camera constants are untouched, and the star-map's determinism and zero-movement guarantees (its seam tests) must still hold. Amber remains only as the `claimed` star. If a status hue needs a token, add a named data-viz token (e.g. `--chart-*` or a small `--status-*` set) rather than sinking a hex into the renderer.

Done when: the terminal renders on a token-derived `ITheme` that visually agrees with the reskinned chrome; the star-map's six states render legibly on the token-derived card surface with the categorical hues intact and amber only on `claimed`; the star-map island-seam tests (determinism, zero-movement on status pushes, selection) still pass unchanged; and `svelte-check`, the Vite build, `vitest`, `go vet`, and `go test` are green.

## Answer

Both islands now source their palette from the shared tokens through the `tokens.ts` bridge (ticket 01), read by each wrapper and handed to the renderer as plain values — neither renderer reaches into CSS itself (ADR 0010).

**xterm (`Terminal.svelte`)** — a `buildTheme()` helper in the wrapper assembles the full `ITheme` at mount: `background`/`foreground` from `--background`/`--foreground` (the terminal sits directly on the page surface, no `Card` wrapper around it, so `--background` rather than `--card`); `cursor` from `--ring` (the app's existing "on/focus" accent, per ticket 03's decisions) with `cursorAccent` back to the surface colour; `selectionBackground` from `--muted`; `black`/`brightBlack` from `--background`/`--muted-foreground` (the "dim" slot the ticket calls out); `red`/`brightRed` both from `--destructive` (no separate "bright" destructive token exists, so the two slots share one value rather than inventing variance). `green`/`yellow`/`blue`/`magenta`/`cyan` and their bright variants have no chrome token to draw from — the theme is otherwise monochrome plus `--destructive` — so those six ANSI slots are literal hex constants, chosen muted enough to sit quietly on the token surface rather than clash with it (contrast-checked at 6:1–10:1 against the resolved `--card`/`--background` hex).

**star-map (`starmap/theme.ts`, `starmap/starmap.ts`, `StarMap.svelte`)** — `starmap.ts` gains a `setBackground(color)` seam method (mirroring `setInsets`); the renderer's hardcoded `g.fillStyle = '#05070d'` now paints whatever's set there, defaulting to the old constant so the existing mount-with-no-args seam tests are unaffected. `StarMap.svelte` calls `sm.setBackground(readColor('--card'))` right after `mount()`. The card moved from a near-black `#05070d` to the theme's lighter warm near-black (`oklch(0.228 0.013 107.4)`, ≈`#1d1d16`) — a real jump in lightness, not a hue-only shift. Checking WCAG-style contrast against that new card: five of the six states still clear 10:1+ on their core colour comfortably; `out_of_scope` — deliberately the dimmest, smallest star — fell to ~3.9:1 core / ~1.8:1 glow / ~4.7:1 label. Its three hex values (and only its three) are lifted just enough to sit legibly again (core, glow, and label each gain roughly one contrast tier); star sizes, glow radii, pulse, and the camera constants are untouched, per the map's re-tune-not-renderer-change instruction. Amber remains only on `claimed`.

**Verified live**: `go run ./cmd/harness` + `npm run dev`, screenshotting the running cockpit (Playwright, headless Chromium — no project `run` skill existed for this repo, so this session drove it directly and the transcript flags `/run-skill-generator` for a future session to capture that path). The star-map card reads as one surface with the surrounding chrome rather than a colder black hole; the terminal's idle background matches the reskinned tab strip above it; printing all 16 ANSI colours in a live shell shows every slot distinct and legible against the token surface.

**Against Done-when.** `svelte-check` — 0 errors/0 warnings. `npm test` — 33/33 (the star-map seam suite — determinism, zero movement across the lifecycle, structure-only relayout, selection/deselection, pane-inset seating — unchanged and green, since palette never touches layout). `npm run build` — clean. `go vet ./...` and `go test ./...` — clean.

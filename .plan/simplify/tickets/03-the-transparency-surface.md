# The transparency surface

## Question

The operator's complaint: the chartr is opaque. Role bindings resolve through three layers nobody can see; prompts materialise into a gitignored directory; map kinds live in a committed TOML file; small preferences have no home at all. The settled direction is **legibility first**: an effective-config surface showing every resolved value with its provenance layer and an open-in-editor hatch, with real editing later and only for high-churn settings. This ticket designs that surface — and takes stock of what "settings" even means after tickets 01 and 02 shrink the system.

This is the first ticket in the repo's history that designs a *screen* whose whole job is explaining the system. The pressure: a settings pane is where good tools go to grow a second config system. The layering rules (ADR 0009) are subtle — workspace wins content, user wins execution, autopilot (dying in ticket 01) was user-only — and a UI that writes TOML can silently violate the very provenance it displays. Legibility-first is a discipline, not a smaller feature.

Settle:

- **What the surface shows.** The full inventory after the cut: role bindings with per-field provenance (built-in/workspace/user) and PATH-probe status, map kinds, resolved skills with their layer, the prompt/payload assembly for a prospective spawn (the preview already exists), ad-hoc preferences. What is missing from that list, and what deliberately is not shown?
- **Where it lives in the cockpit.** A settings pane per space, a global one, or both? The cockpit has no router beyond a hash deep-link and no settings screen at all — does this earn a first-class surface, and what is the navigation model?
- **The read path.** `config.Resolve` already computes provenance — is exposing it a handler away, or does the resolution need restructuring to be reportable? Skills and map kinds need the same treatment; is there one "effective state" endpoint or several?
- **The edit boundary.** Which settings are high-churn enough to earn real editing later (role bindings are the named candidate), and what is the write discipline when that lands — structured TOML edits preserving comments and operator formatting, like `DeclareMapKind` already does? What must the UI *never* write (committed workspace config from a local UI is the obvious trap)?
- **Explaining the flow.** The operator's deeper complaint is that *how the whole thing works* is invisible. Does the surface carry explanation (inline "why" text, a diagram, links into docs), or is that a docs problem the UI should stay out of? Where does a confused user's first five minutes actually go?

## Answer

**The transparency surface is a first-class global settings route — the cockpit's first real route — that renders every space's resolved config from the model the client already receives, explains each value by its provenance layer in place, opens the underlying file for anything it does not itself edit, and edits the one high-churn thing (role bindings) inline, writing only that value's home layer.** It is legibility-first as a discipline: it shows resolved values with their provenance and refuses to become a second config store — no invented preferences table, no form over TOML the files already own. Almost all of its read path exists today; the new work is a route, one derived skills field, two narrow write handlers, and the screen itself. This **writes a new ADR (0014 — the transparency surface)** and **amends ADR 0009** with an edit-boundary consequence; it inherits ticket 01's cut (autopilot is gone, so there is nothing to toggle) and ticket 02's skill library (skills resolve by whole-skill shadowing, with `forked_from` drift).

### What the surface shows

The full inventory after the cut, per space, plus the one truly-global file:

- **Role bindings** — each role's effective `{adapter, model, args?}` with **per-field provenance** (built-in / workspace / user) and **PATH-probe status**. This is exactly what `config.Resolved` already computes (`AdapterFrom`/`ModelFrom`/`ArgsFrom`, `Present`/`Missing`) and what the existing per-space bindings Sheet (`SpacePane.svelte`) already renders — generalised, and now editable (below).
- **Resolved skills** — after ticket 02, each skill (`core`, `tracker-convention`, and the role skills) with the **layer that won its whole directory** (whole-skill shadowing) and its **stale-fork** state (`forked_from` behind the shipped hash). This is the skill equivalent of a binding's provenance line; today only the *warnings* leak out (`prompt.LibraryWarnings`, already folded into `space.warnings`), never the positive "your `grill` skill resolves from: user."
- **Map kinds** — the committed `slug → kind` declarations (`Resolution.Kinds`), read-only here because kind is edited through the existing **classify** action (committed, human-confirmed, ADR 0007); the surface shows it and links to that action rather than duplicating it.
- **Config file locations** — the concrete path of each layer that participated: the space's `.chartr/config.toml`, the user file under `~/.config/chartr/`, and the data-dir skills root. Naming where each layer *lives* is itself legibility, and the open-in-editor hatch needs the path anyway. (Added to the ticket's list — it was implied, not stated.)
- **Warnings** — the resolver's surfaced notices (malformed file, unknown role, unrecognised kind) already on `space.warnings`.
- **The prospective-spawn payload** — reached, not rebuilt: the surface links to the existing payload preview (`PayloadPreview.svelte` / `GET …/payload`), which is parameterised by a prospective `(role, ticket)` and stays its own on-demand fetch.

**Deliberately not shown:** runtime session state (the star-map and terminal own liveness, claims, quiet hints — config is not the place for it); **secrets or tokens** (the chartr holds none — each adapter owns its own auth, and a settings pane that grows a credentials box is a new attack surface for zero chartr value); and the **raw TOML re-rendered as a form** for everything — only bindings get an editor; every other value is a resolved value plus an open-the-file hatch, because the files stay the interface (the map's settled decision).

**Deliberately no home for "ad-hoc preferences."** With autopilot deleted by ticket 01, the cut leaves no real ad-hoc preference behind. Inventing a preferences table now — a KV store the settings pane reads and writes — is precisely the "second config system" the ticket warns a settings pane grows. Refused: the surface shows what the three documented layers resolve, and nothing else. If a genuine preference ever appears, it earns a layer and a home then; it does not get one speculatively.

### Where it lives, and the navigation model

**A global settings route — the cockpit's first real route — not a per-space Sheet.** The read path is per-space (bindings, skills, kinds all resolve per space), but the *home* is one global screen that lists every space and the one global user file, because the operator wanted a first-class surface rather than a drawer bolted to one space's header, and because the user config is a single file spanning all spaces that no per-space drawer can honestly own.

- **The router is a hash prefix, not a routing library.** The cockpit already parses `location.hash` for the star deep-link (`#s=…&m=…&t=…`, `SpacePane.svelte`). The settings route rides the same mechanism: a `#/settings` prefix (`#/settings/s=<spaceId>` to focus a space, `#/settings/user` for the global file view) that `App.svelte` reads in one `$derived` route and, when present, renders `<Settings/>` in place of the space cockpit. ~15 lines, no dependency — a real route, but the thinnest thing that is one. The bare star deep-link is untouched (it has no `/` prefix), so the two schemes never collide.
- **Entry and exit.** A **⚙ button in the sidebar header**, beside the existing `Bell` queue trigger, toggles the route; **`,`** (the conventional preferences key) does the same from the keyboard, alongside the existing `[`/`]`/`q`/`M`; **Esc** or selecting any space returns to the cockpit. The screen itself is a left rail of spaces (plus a "User config — all spaces" entry) and a main panel of the selected space's sections.
- **The existing per-space bindings Sheet folds in.** `SpacePane.svelte`'s "bindings" button becomes a "config" button that **navigates to `#/settings/s=<thisSpace>`** rather than opening a local Sheet — one config home, not two. The knowingly-accepted cost: a quick binding peek now costs a route change instead of a drawer; it is one keystroke, and one home beats the two-surfaces drift the global choice exists to avoid. (Revisit: if the peek is missed, a read-only hover-card on the space header returns — deep-linking into settings for anything more.)

Rejected: **both per-space and global** (two homes for config to grow in — the exact multiplication the ticket flags); **a routing library** (a route this shape does not earn a dependency the offline bundle would carry).

### The read path

**One effective-state channel already exists — the per-space model push — and the surface renders from it; almost nothing new is a "handler away" because it is already pushed.** `Server.deriveSpace` already calls `config.Resolve` per space on every rebuild and folds `Bindings` (full provenance + PATH probe), `Kinds` (onto each map), and `Warnings` into the `model.Space` the client holds. So the surface needs **no new read endpoint for bindings, kinds, or warnings** — it re-renders state `App.svelte` already receives over `/ws/control`.

- **One new derived field: resolved skills.** Ticket 02's skill resolver already has to compute which layer wins each skill; reporting that (name → winning `Layer`, plus `forked_from`/stale) is the same one-line addition that `Resolved`'s `*From` fields already are for bindings. It rides the model push as a new `Space.Skills []ResolvedSkill`, derived in `deriveSpace` next to the bindings loop. This is the only resolution work that needs *exposing* rather than just *rendering*; it does not need `config.Resolve` restructured — it needs ticket 02's resolver to return its provenance the way `config.Resolve` already returns bindings'.
- **The payload preview stays separate and on-demand.** It is parameterised per prospective `(role, ticket)` and would be wrong to push for every pair; the surface links to the existing endpoint, it does not absorb it. So "one effective-state endpoint or several" resolves to: **the model push is the one channel for durable state; the payload preview is the one on-demand fetch; the open-in-editor hatch is one new action** — not a monolith, and not a scatter.

### The edit boundary

**Role bindings are editable inline now; everything else is read-value-plus-open-file.** Bindings are the named high-churn candidate and the operator chose to ship their editing with the surface rather than defer it — so the discipline the map warned about ("editable second … must never become a second source of truth against the layering") lands now, not later, and the answer is precise about it:

- **Binding edits write the user layer only** — `~/.config/chartr/config.toml`, `[spaces."<path>".roles.<role>]`. Never committed workspace config. This is not a limitation, it is the layering: bindings resolve **user-over-workspace** (ADR 0009 — execution choices the operator makes win), so the *correct* target for one machine's binding change **is** the user layer. A local UI writing committed workspace bindings would silently override shared content the surface itself labels "workspace" — the exact provenance violation a config screen must never commit. **The UI writes a value only into that value's own home layer.**
- **The write is surgical and comment-preserving — and harder than `DeclareMapKind`.** `DeclareMapKind` only ever *appends* a fresh `[maps."<slug>"]` table to an absent slug, so it never has to touch existing bytes. A binding edit must *update* a table that may already exist, and a naive decode-and-re-encode would flatten the operator's comments and formatting — the precise hackability violation the append-only stance exists to prevent. So the write is a **key-level edit**: set or clear the specific `adapter`/`model`/`args` keys within the target role table (creating the table if absent, appending in `DeclareMapKind`'s style), leaving every surrounding byte — comments, ordering, unrelated tables — intact. This is real implementation cost, and it is *why* the map framed editing as the harder half.
- **Editing is reversible.** Clearing a field's override deletes that user-layer key, revealing the layer beneath (workspace, then built-in) — the provenance badge flips back accordingly. The operator can always return to the shipped default; editing is not a one-way ratchet into an ever-growing user file.
- **After a write, the model re-derives.** The write handler triggers the same rebuild the classify handler does, so the edited value and its new provenance reflect straight back into the surface over the control socket — no optimistic client state to drift from the file.
- **What the UI must never write:** committed workspace config for *execution* values (bindings) from a local machine — refused above. Map **kind** stays a committed-layer value written only through the deliberate, human-confirmed **classify** action (ADR 0007), never through this editor. And there is no autopilot toggle to get wrong — ticket 01 deleted the flag.
- **Everything not editable gets the open-in-editor hatch.** Each layer's row carries an "open the file" action → a new narrow `POST …/config/open` that launches `$VISUAL`/`$EDITOR`, falling back to the OS opener (`open`/`xdg-open`), and finally to surfacing the absolute path to copy (the spawn-block pattern — name what is missing, don't paper it over). The handler opens a **named layer file resolved server-side** (workspace / user / a named skill dir), never a client-supplied path — a local server that opens arbitrary paths on request is a trap the enumeration closes. This works identically under the webview shell (ticket 04) since the server is local either way.

### Explaining the flow

**The surface carries light, in-place "why," and links out for the method — it does not restate how the system works.** The operator's deeper complaint is answered by making provenance *visible*, not by growing a tutorial that rots against the thing it explains:

- **Provenance badges are the explanation-in-place.** "model ← workspace, adapter ← built-in" *is* the why, rendered where the value is. Each section also keeps a one-line layering caption (the bindings Sheet already has one: "merged built-in ‹ workspace ‹ user; the tag names the layer it came from").
- **Deep "how it works" links into skills and docs, which now own it.** After ticket 02 the `core` and `tracker-convention` skills are on-disk, hackable markdown that already *are* the "how to use this chartr" and "the method" explanations; the surface links a "How resolution works →" into them and into `docs/adr/0009`, rather than duplicating prose that would drift from the canonical source. The UI stays out of the docs business precisely because ticket 02 gave the docs a legible home.
- **The first five minutes.** The surface answers "**why is this value what it is, and where do I change it**" — the config questions — and hands "**what is a wayfinder map / what is the lifecycle**" to the skills and docs it links. A settings screen that tried to teach the method would be a worse teacher than the skills and a permanent maintenance tax; keeping it to provenance-and-locations is the legibility-first line.

Rejected: **an in-app diagram or tutorial of the flow** (a docs problem the UI should not own — it rots out of sync with the skills that are now the canonical, hackable explanation, and it is exactly the scope creep that turns a legibility surface into a second product).

### What this writes

- **New ADR 0014 — the transparency surface (legibility first):** a global settings route as the cockpit's first real route (hash-prefix, no library); the effective-state read path riding the existing per-space model push plus one derived skills field; inline editing scoped to role bindings and to the user layer only, with comment-preserving key-level writes and reversible overrides; the open-in-editor hatch over server-resolved layer files; the refusal of a preferences store; and the method-explanation left to the skills/docs the surface links.
- **ADR 0009 — amended.** The layering *mechanism* is untouched (the map kept it out of scope). Two consequences are added: its **surface** is now this effective-config screen (the field-level "render the effective binding" consequence it already anticipated, made concrete); and the **edit-boundary rule** — a UI writes a value only into that value's home layer (bindings → user, kind → workspace via classify) — is recorded so a future editor cannot quietly violate the provenance the same ADR defines. The autopilot bullets are already gone (removed by ticket 01), so this ticket does not re-touch them.
- **`CONTEXT.md`** — a new **Effective config surface** term (the global screen showing every resolved value with its provenance and file location; the home for legibility). The **Prompt library → Skill library** rename is ticket 02's; this ticket only relies on it for the skills section.
- **Code (for the implementation map, not built here):** a `Settings.svelte` route + `App.svelte` route derivation and ⚙/`,` entry; `SpacePane.svelte`'s bindings button rewired to navigate; a `Space.Skills` field in `internal/model` derived in `deriveSpace`; a user-layer binding writer in `internal/config` (key-level, comment-preserving) behind a `PUT …/config/binding` handler that rebuilds; and a `POST …/config/open` editor-launch handler over server-resolved layer paths.

### Revisit trigger

Two tripwires, either sufficient:

- **A second editable setting earns editing.** If any value beyond bindings becomes high-churn enough that open-in-editor is a genuine friction — not an anticipated one — it earns an inline editor built to the same rule (write only its home layer, comment-preserving, reversible). Until then, open-the-file is the honest surface for the low-churn rest.
- **The global route is felt as a detour.** If routing away from a space to read its own config proves to break flow in daily driving, the per-space quick-peek returns as a read-only hover-card on the space header that deep-links into `#/settings` for editing — additive, not a second config home.
